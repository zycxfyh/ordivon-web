Publishing interface · Static delivery · Active refinement

Explain evolving projects without replacing the truth that owns them.

Ordivon Web is the editorial and navigation layer around independent engineering work. It introduces the questions, methods, evidence, changes, and open problems behind each project while routing readers back to repositories, source, tests, release records, and current documents.

Central question

What should a public project page contain when the underlying system keeps changing?

A repository is precise but expensive to reconstruct. A conventional marketing page is easy to scan but often removes uncertainty, history, and boundaries. Ordivon Web tries to occupy the useful middle: enough editorial structure to understand the work, enough links and dates to verify it, and enough restraint to avoid becoming a second mutable registry.

The page is successful when a visitor can identify the project question, present state, evidence, evolution, and next uncertainty—then reach the canonical source without learning the repository’s entire internal vocabulary.

Evolution

The site became more useful by adding editorial structure, then removing unrelated scope.

  1. Initial site
    Establish a stable public address.

    A small static page introduced Ordivon and proved deployment without a framework or backend.

  2. Multi-page foundation
    Add Projects, Notes, Current, About, Contact, metadata, and verification.

    The site gained enough information architecture to support independent project and article routes.

  3. 22–23 Jul 2026
    Publish Runtime evidence and move the canonical domain to the apex.

    Release records, project depth, Cloudflare DNS, www routing, and lab redirect became part of the public operating surface.

  4. PR #15 onward
    Reframe the site as a personal research and engineering interface.

    Commercial-product drift and unrelated portfolio scope were removed; project questions, evidence, and editorial hierarchy became primary.

Open questions

The next improvements concern depth, return visits, and sustainable verification.

How much project detail belongs on the site before it begins duplicating repository documentation?

Which note metadata and related-reading links meaningfully help return visitors without requiring a CMS or premature taxonomy?

Which browser, accessibility, and performance checks catch real presentation failures without creating a heavy frontend toolchain?
