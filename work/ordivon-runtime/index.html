Agent infrastructure · Trusted-local Runtime · Adoption

Preserve the execution facts an agent cannot reconstruct.

Ordivon Runtime sits between an MCP client and the local host. The agent continues to reason, choose tools, and write implementation; Ordivon owns the narrow execution identity, process tree, bounded output, cancellation intent, result, and Artifact state required after interruption.

Central question

What must remain true when a capable agent loses its client connection?

A conversation can reconstruct a plan. It cannot reliably reconstruct whether a process started, which command tree still owns the host, whether cancellation was requested, which bytes were retained, or whether a later result belongs to the original execution identity.

Ordivon exists to preserve those non-reconstructable facts without replacing the agent loop, Git, the repository, mature host CLIs, or the operating system.

Evolution

The project became smaller as the problem became clearer.

  1. Early direction
    Govern AI work through a broad platform.

    The project accumulated policy, registry, proof, document, and workflow concepts before real use established which facts had to be durable.

  2. M0–M7
    Build and test an execution core.

    Workspace identity, transactional execution, process ownership, cancellation, bounded output, Artifacts, and recovery became the concrete center.

  3. 22–23 Jul 2026
    Delete the speculative platform around the core.

    The active documentation, protocol, deployment, and repository were reduced to one trusted-local Runtime path.

  4. Current
    Let real tasks decide what remains missing.

    New capabilities require repeated operational failure that mature host tools through workspace.exec cannot solve.

Open questions

The remaining uncertainty is operational, not a request for another platform layer.

Which real tasks still require tools outside the ten-tool surface?

How reliably do backup, restore, restart reconciliation, and cancellation behave under repeated long-running work?

Where should users place an external VM or container boundary when trusted-local authority is too broad?
