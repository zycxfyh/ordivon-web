About

A public interface for projects that change through use and inquiry.

Ordivon brings together independent engineering work, authored notes, release records, and current questions. The aim is not to present a finished company or a fixed theory, but to make the development of difficult projects easier to understand, inspect, and revisit.

Purpose

Publish the reasoning around the repositories, not a second copy of them.

Source code, tests, data, Git history, and current-state documents remain the owners of technical facts. This site provides the editorial layer: why a project exists, which question it addresses, how the approach changed, what has been demonstrated, and what remains open.

That distinction keeps the public explanation useful without forcing every visitor to reconstruct the project from source—or allowing prose to silently replace live system truth.

Connected questions

How can capable systems remain understandable while they evolve?

Ordivon Runtime asks which execution facts must survive interruption when agents operate real machines. FinHarness asks how capital state, uncertainty, judgment, bounded action, and later review can remain connected. Ordivon Web asks how work like this can be presented honestly without becoming a marketing abstraction or a duplicate registry.

The projects share methods and lessons, but each remains independently understandable and keeps its own domain boundaries.

Working method

Questions move through research, construction, failure, and revision.

01Find the real problem

Begin with observed friction, uncertainty, failure, or a decision that existing abstractions do not explain well.

02Inspect existing knowledge

Read the relevant research, standards, systems, source, data, and prior attempts before inventing another layer.

03Build a bounded model

Define the objects, mechanisms, claims, and non-goals needed for the current problem.

04Test through real work

Use implementation, analysis, dogfood, counterexamples, and review to expose where the model fails.

05Revise and publish

Change the system, record the result, and preserve unresolved questions rather than forcing premature closure.
